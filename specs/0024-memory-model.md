# 0024: Memory Model

Status: Draft

ヒープ値（0001）のメモリ管理モデルを定義する仕様．**ヒープグラフの非巡回性**を言語の規範的
不変条件として固定し，WASM/WAMR ターゲットで **wasm-gc に依存しない** 自前ヒープと，
**決定的な参照カウント (RC)** による完全な回収を可能にする．

## Summary

- Emela のヒープグラフは**非巡回 (acyclic)** である．これは言語仕様が保証する規範的不変条件である．
- 回収（reclamation）はプログラムから**観測不能**である．観測可能なのは資源枯渇（alloc 失敗）のみで，
  それは panic（0011）とする．
- WASM/WAMR ターゲットでは，ヒープは linear memory 上のランタイム自前実装とし，**wasm-gc proposal に
  依存してはならない** (MUST NOT)．
- 参照実装は**決定的 RC**．非巡回性により cycle collector なしで完全（リークしない）である．
- JS ターゲットはホスト GC を使ってよい（回収は観測不能なので観測可能な意味は一致する）．

## Motivation

0001 は「基本的に GC に近い仕組みでメモリを管理する」とだけ定めており，方式は未定だった．WAMR
（組込み・エッジ）を一次ターゲットとする以上（0000 原則6），次の制約がある．

- WAMR の wasm-gc サポートは未成熟であり，依存できない．よって linear memory 上の自前ヒープが必要．
- 組込み環境では GC ポーズと非決定的なメモリ挙動が問題になる．決定的な回収は直接の価値である．

ここで Emela の既存仕様が効く: 全ヒープ値は immutable（0001, 0002, 0007）．`let` は非再帰（逐次束縛，
0002）．再帰はトップレベル関数に限られ，トップレベル関数の参照は静的で environment を持たない（0003）．
無名関数が capture できるのは定義時点で既に存在する値だけである．したがって**ヒープ参照は常に
「後から作られた値 → 先に作られた値」の向きにしか張れず，巡回は構成できない**．この性質を規範化すれば，
純粋な RC が完全（サイクルによるリークなし）かつ決定的な回収戦略になる．

## Specification

### 観測可能性

- メモリ管理はプログラムの観測可能な挙動に影響してはならない (MUST)．finalizer・デストラクタ・
  weak reference のような，回収を観測する手段は存在しない (MUST NOT)．
- 割り当ての失敗（ヒープ枯渇）は panic である（0011 と同様，回復不能．通常評価へ復帰しない）．

### 非巡回性（規範的不変条件）

- 言語のいかなる評価も，ヒープ参照の巡回を構成できてはならない (MUST)．現行仕様でこれは次から帰結する:
  1. すべてのヒープ値は構築後 immutable である（0001）．
  2. `let` 束縛は非再帰であり，右辺は束縛対象自身を参照できない（0002）．
  3. 再帰はトップレベル関数に限られ，その参照は静的である（closure の env を経由しない，0003）．
- 巡回を構成しうる機能（再帰的 `let` / 可変セル / 遅延評価とメモ化 / 自己参照 closure 等）を導入する
  将来仕様は，本仕様を supersede しなければならない (MUST)．

### WASM / WAMR ターゲット

- ヒープは linear memory 上にランタイムが実装する (MUST)．wasm-gc proposal の型・命令に依存しては
  ならない (MUST NOT)．
- 生成されるモジュールは，interpreter / AOT のいずれの WAMR 実行モードでも同一の観測可能な意味を
  持たなければならない (MUST)．

### 回収戦略（参照実装）

- 参照実装は参照カウントとする．非巡回性（上記）により，RC は完全である（到達不能な値は必ず回収され
  る）．cycle collector は不要である．
- 実装は，最後の参照が消滅した時点で決定的に回収すべきである (SHOULD)．これにより GC ポーズが存在せず，
  メモリ使用量が実行ごとに決定的になる．
- ただし回収は観測不能であるから，これと異なる戦略（tracing GC，ホスト GC，arena 等）を取る実装も適合
  する (MAY)．JS バックエンドはホスト GC を使ってよい．

## Examples

```emela
fn make() -> Array<Int> {
    let xs = [1, 2, 3]      -- alloc
    let ys = [4, 5, 6]      -- alloc
    ys                      -- xs はここで到達不能 → RC 実装では決定的に回収される
}
```

回収のタイミングはプログラムから観測できないため，この「回収される」は実装保証（資源挙動）であって
意味論ではない．

## Compilation Notes

この節は非規範的である．

- **ヘッダ**: ヒープ値は `[refcount: i32][kind/tag: i32][payload...]` 程度の小ヘッダを持てばよい．
  `String` の `[len][utf8 bytes]`（0013/0017）はヘッダに scalar 数のキャッシュを足すと `String.length`
  （0030）が O(1) になる．
- **RC 操作の挿入**: typed IR（0012）上で束縛・引数渡し・戻り・スコープ脱出に対して incref/decref を
  挿入する．immutable なので所有権解析による省略（借用と同型の最適化）が安全に効く．
- **アロケータ**: bump allocator + free list 程度で足りる．決定的回収により断片化が問題になれば
  size-class 化する．
- **Native バックエンド**: 同じ RC ランタイムを共有できる．
- **WAMR 埋め込み**: モジュールの memory は自己完結し，ホストとの値の受け渡しは 0013 のマーシャリング
  境界を通る．ホストが linear memory を直接参照する場合の規約は 0025 / backend ABI に属する．

## Open Questions

- ヒープサイズの上限設定・OOM panic のメッセージ / フックの規約．
- incref/decref 省略最適化（静的所有権解析）の非規範ガイドライン．
- 将来 async / 継続を入れる場合（0000 は非目標）のスタックと RC の相互作用．
- ホストへ渡した参照の寿命（0013 のマーシャリング ABI と共有: コピー渡しなら問題は生じない）．
