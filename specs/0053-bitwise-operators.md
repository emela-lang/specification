# 0053: Bitwise Operators and Shifts

Status: Draft

`Int` に対するビット演算子 `& | ^ ~` とシフト演算子 `<< >> >>>` を定義する仕様。0027 が Open Question（0027:112-113）として先送りしていたビット演算を実体化する。二項ビット演算子は 0020 のトレイト演算子として定義し（`+` と同じ機構）、単項 `~` は新トレイトを導入せず `^` へ導出 desugar する。あわせて 0027 の統合優先順位表を、ビット演算・シフトを組み込んだ形に改訂する。

## Summary

- 二項演算子 `&`（AND）・`|`（OR）・`^`（XOR）・`<<`（左シフト）・`>>`（算術右シフト）・`>>>`（論理右シフト）を追加する。いずれも 0020 のトレイト演算子であり、`Int` の instance を Core Prelude が与える。
- 前置演算子 `~`（ビット反転）を追加する。新しいトレイトは導入せず、`~e ≡ e ^ -1` へ導出 desugar する（0027 の `a != b ≡ !(a==b)` と同じ方式）。
- ビット演算・シフトはいずれも **`Int` 専用**である。`Float` / `Bool` の instance は与えない。
- 0027 の全演算子優先順位表を、ビット演算・シフトを含む形に改訂する。

## Motivation

0027:112-113 は「ビット演算（`& | ^ << >>`）の導入時期」を Open Question として残していた。ビット単位の操作は、擬似乱数生成器（xorshift 系、0054）・ハッシュ・ビットフラグ・低レベルなバイト処理（0051）などで必要になる。とりわけ 0054 のシード可能 PRNG は XOR と論理右シフトを要し、これらが無いと純 Emela では書けない。

`Int` は 0001 で `i32`（32bit・2の補数・符号付き）に固定され、算術は mod 2^32 で wrap する（0001:67）。ビット演算・シフトはこの 32bit 表現の上で定義され、すべてのバックエンドで同一の結果を与える（0000: ターゲット間で観測可能な意味が変わらない）。

`>>`（算術）と `>>>`（論理）を分けるのは、符号付き `Int` に対して符号拡張シフトとゼロ拡張シフトの両方が必要になるためである（Java / JavaScript と同じ区別）。

## Specification

以下、キーワードは RFC 2119 に従う。対象の整数表現は 0001 の `Int`（i32）である。

### 二項ビット演算子（トレイト演算子）

次の演算子を追加する。いずれも 0020 のトレイト演算子であり、両オペランドと結果の型は同一である（MUST）。

| 式 | トレイト | メソッド | `Int` 上の意味（i32） |
| --- | --- | --- | --- |
| `a & b`  | `BitAnd` | `bitand` | ビット単位 AND |
| `a \| b` | `BitOr`  | `bitor`  | ビット単位 OR |
| `a ^ b`  | `BitXor` | `bitxor` | ビット単位 XOR |
| `a << b` | `Shl`    | `shl`    | 左シフト（下位にゼロ充填） |
| `a >> b` | `Shr`    | `shr`    | 算術右シフト（符号ビットを拡張） |
| `a >>> b`| `UShr`   | `ushr`   | 論理右シフト（上位にゼロ充填） |

- **B1（Int 専用）**: Core Prelude は上記 6 トレイトの instance を **`Int` に対してのみ** 与える（MUST）。`Float` / `Bool` への instance は与えない（MUST NOT）。したがって `Float & Float` 等は instance 不在で型エラーになる（`Float % Float` が `impl Rem for Float` 不在でエラーになるのと同じ、0016 と整合）。ユーザー型に `impl BitAnd` 等を与えれば `&` 等を使えるようにしてよい（0020）。
- **B2（シフト量）**: `<<` / `>>` / `>>>` のシフト量 `b` は `Int` である。ハードウェア慣行に従い、実効シフト量は `b` の下位 5 ビット（`b & 31`）とする（MUST）。これは WASM の `i32.shl` / `i32.shr_s` / `i32.shr_u` および JavaScript のシフトのネイティブな意味と一致する。負の `b` や `b >= 32` はこの mask により定義され、trap しない。
- **B3（結果の一意性）**: 各演算はビットパターン上で一意に定まり、trap も panic もしない（MUST）。`>>` は最上位ビット（符号）を拡張し、`>>>` はゼロを充填する。`<<` の溢れたビットは捨てられる（mod 2^32、0001:67 と整合）。
- **B4（effect）**: いずれも 0020 の通常のトレイトメソッド呼び出しであり、`effects(a op b) = effects(a) ∪ effects(b)`。トレイトメソッド本体は `uses {}`。

### 前置演算子 `~`（導出）

- **U1**: `~e` は前置演算子である。次の等価が規範である（MUST）: `~e ≡ e ^ -1`。`-1`（i32 で全ビット 1）との XOR により全ビットを反転する。新しいトレイトは導入せず、`^`（`BitXor`）へ desugar する。オペランド型に対する要求は `^` と同一である（`BitXor` の instance が in-scope に必要）。したがって `~` は実質 `Int` 専用である。
- **U2**: desugar は構文変換であり、`e` の評価回数・評価順を変えない（MUST）。

### 優先順位（統合表・0027 改訂）

強い順に、次を規範とする（MUST）。0027:56-68 の表を、シフト（4）とビット演算（5-7）を挿入する形に改訂する。

```text
1.  ! ~                  (前置)
2.  * / %                (左結合, 0016)
3.  + - ++               (左結合, 0017)
4.  << >> >>>            (左結合)
5.  &                    (左結合)
6.  ^                    (左結合)
7.  |                    (左結合)
8.  == != < > <= >=      (左結合, 0027)
9.  &&                   (左結合, 0027)
10. ||                   (左結合, 0027)
11. |>                   (左結合・最低, 0019)
```

- シフト（4）は算術（2, 3）より弱い: `a + b << c` は `(a + b) << c`。
- ビット演算（5-7）は比較（8）より **強い**（MUST）: `x & 1 == 0` は `(x & 1) == 0` と解釈される。これは C 系の「比較がビット演算より強い」順序を意図的に採らない（`x & (1 == 0)` という誤読を避ける。Python と同じ順序）。
- ビット演算どうしは `&`（5）> `^`（6）> `|`（7）の順に強い: `a | b & c` は `a | (b & c)`、`a ^ b & c` は `a ^ (b & c)`。
- `~` は前置で最も強い（1）: `~a & b` は `(~a) & b`。

### 字句（非規範の補足を含む）

- `&` は単独で `BitAnd`、`&&` は論理積（0027）である。`|` は単独で `BitOr`、`||` は論理和（0027）、`|>` はパイプ（0019）である。字句解析は最長一致（`&&` / `||` / `|>` を単独 `&` / `|` より優先）とする（MUST）。
- `<<` / `>>` / `>>>` は、`<` / `>`（比較、0027）およびジェネリクスの `<` / `>`（0028）と字句を共有する。ネストしたジェネリクス（例 `Array<Array<Int>>`）の閉じ `>>` を壊してはならない（MUST NOT）。実装はシフトを型構文と衝突しない形で解析しなければならない（実装ノート参照）。

## Examples

```emela
fn mask(x: Int) -> Int uses {} {
    x & 255                       -- 下位 8 ビット
}

fn set_flag(flags: Int, bit: Int) -> Int uses {} {
    flags | (1 << bit)
}

fn is_even(x: Int) -> Bool uses {} {
    x & 1 == 0                    -- (x & 1) == 0
}

fn high_bits(x: Int) -> Int uses {} {
    x >>> 16                      -- 上位 16 ビット（ゼロ拡張）
}

fn invert(x: Int) -> Int uses {} {
    ~x                            -- x ^ -1
}
```

xorshift32 の 1 ステップ（0054 のシード可能 PRNG が使う形）:

```emela
fn step(x: Int) -> Int uses {} {
    let a = x ^ (x << 13)
    let b = a ^ (a >>> 17)        -- 論理右シフト（ゼロ拡張）
    b ^ (b << 5)
}
```

## Compilation Notes

この節は非規範的である。

- 各二項演算子は 0020 のトレイト解決に乗り、組み込み `Int` では intrinsic（0021）へインラインされる: `i32_and` / `i32_or` / `i32_xor` / `i32_shl` / `i32_shr_s`（算術）/ `i32_shr_u`（論理）。WASM ではそれぞれ `i32.and` / `i32.or` / `i32.xor` / `i32.shl` / `i32.shr_s` / `i32.shr_u`、JavaScript では `&` / `|` / `^` / `<<` / `>>` / `>>>`（`>>>` は結果を `| 0` で i32 表現へ戻す）へ lower する。
- `~e` は `e ^ -1` へ desugar されるため、`BitXor` の経路をそのまま通る。専用の intrinsic は不要。WASM に `i32.not` は無く、`-1` との `i32.xor` で表現する。
- **シフトとジェネリクスの字句衝突**: `Array<Array<Int>>` の閉じ `>>` は 2 個の `>` として型構文で消費される。実装は `<` / `>` を単一トークンに保ち、式のシフト解析でのみ隣接（スパン連続）する `< <` / `> >` / `> > >` を左シフト / 算術右シフト / 論理右シフトへ合成することで、型構文（0028）を無変更に保てる。`>=` は単一トークンなので合成に混入しない。

## Open Questions

- 複合代入（`&=` `|=` `<<=` 等）: Emela は不変束縛（0002）主体であり、当面導入しない。
- `Bool` に対する `&` / `|`（非短絡のビット論理）: 論理は `&&` / `||`（0027）で足りるため、当面 `Int` 専用に留める。
- 回転（rotate）や `count_ones` / `leading_zeros` 等のビット操作関数: 必要になれば stdlib 関数（0021 intrinsic）として別途導入する。

## 他仕様との関係

- **0001（Core Values and Types）**: ビット演算・シフトの対象は `Int`（i32、mod 2^32）である。
- **0020（Traits）**: 二項ビット演算子を 6 つのトレイト演算子として追加する（`+` 等と同じ機構）。
- **0021（Intrinsics）**: `Int` の instance が落ちる先の intrinsic（`i32_and` 等 6 個）を追加する。
- **0027（Boolean and Comparison Operators）**: 統合優先順位表を改訂し、Open Question（0027:112-113）のビット演算を解決する。`&` / `&&`、`|` / `||` / `|>` の字句上の棲み分けを定める。
- **0028（Generic Data Types）**: `<<` / `>>` の字句がジェネリクスの `<` / `>` と衝突しないことを要求する。
- **0054（Random）**: シード可能 PRNG（xorshift32）が本仕様のビット演算・論理右シフトを用いる。
