## 0045: Self Tail Calls — 自己末尾呼び出しの保証

Status: Draft

自己末尾呼び出しのスタック不消費を言語保証とする（2026-07-19 決定）．

トップレベル関数の**直接自己再帰**が末尾位置にあるとき，その呼び出しはスタックを消費しないことを保証する仕様．0003 が「末尾再帰呼び出しが最適化されるかどうかは規定しない」と明示的に空けた点を，自己呼び出しに限って規定する．Emela にはループ構文がなく（0002・0004——全束縛不変の式指向），無限に回る処理（HTTP サーバーの accept ループ，0046）は再帰でしか書けないため，この保証がないと長時間稼働プログラムを書けない．

### Summary

- 関数 `f` の本体の**末尾位置**にある `f` 自身の直接呼び出しは，スタックを消費しない（MUST）．自己末尾再帰の深さは実行時間・メモリ以外の資源で制限されない．
- 末尾位置は構文的に定義する：本体 block の最終式，末尾位置にある `if` の両 branch・`match` / `catch` の各 arm の最終式．`try` block の内部・`?` の対象・引数位置は末尾位置ではない．
- これは最適化（任意）ではなく保証（規範）である．観測可能な意味は通常の呼び出しと同一である．
- 相互再帰・関数値経由の呼び出しは対象外である（Open Questions）．

### Motivation

1. **無限ループの表現手段**．Emela は全束縛不変（0002）の式指向言語であり，`loop` / `while` を持たない．反復は再帰で書く（0003）．しかし 0003 は末尾再帰の最適化を規定しないため，無限に回るループ（サーバーの accept ループ，イベント処理）は「いつかスタックが溢れる」プログラムになってしまう．0046（HTTP Server）は `serve_loop(server)` 形の自己末尾再帰を規範例に持つ最初の仕様であり，この保証を前提とする．
2. **ループ構文よりも言語の性格に合う**．`loop` 式の導入は，全束縛不変の下で「周回をまたいで状態を持ち回す」手段（可変束縛・`break` の値）を芋づる式に要求する．自己末尾再帰は引数がそのまま「次の周回の状態」であり，既存の意味論（0002 / 0003 / 0004）に一切手を入れずに反復を表現できる．
3. **実装は局所変換で足りる**．直接自己呼び出しに限れば，各 backend は関数内のループへの変換（引数の再束縛 + 先頭への分岐）で実装でき，WASM の tail-call 拡張命令にも，スタック切り替えにも依存しない（0000 の WAMR 制約と整合）．保証を相互再帰まで広げると tail-call 命令への依存が生じるため，本仕様は自己呼び出しに限定する．

### Specification

以下，キーワードは RFC 2119 に従う．

#### 末尾位置（T 規則）

- **T1（定義）**: 式の**末尾位置**を次のとおり帰納的に定める：
  - 関数本体 block（0004）の最終式は末尾位置である．
  - 末尾位置にある block 式の最終式は末尾位置である．
  - 末尾位置にある `if` 式（0015）の `then_block` / `else_block` の最終式は末尾位置である．
  - 末尾位置にある `match` 式（0005）の各 arm 本体の最終式は末尾位置である．
  - 末尾位置にある `try` / `catch` 式（0011）の各 **catch arm** 本体の最終式は末尾位置である．**`try` block の内部は末尾位置ではない**（error の捕捉が未完了のため）．
  - 上記以外——束縛の右辺・引数位置・演算子の被演算子・`?` の対象（0011——error 経路の後処理が残る）・pipeline（0019）の左辺——は末尾位置ではない．
- **T2（保証）**: トップレベル関数 `f` の本体の末尾位置にある，`f` 自身への直接呼び出し（ベア名 `f(...)` による同一関数の呼び出し）は，スタックを消費してはならない（MUST）．すなわち自己末尾呼び出しの列は，深さによらず一定のスタックで実行される．
- **T3（意味の不変）**: T2 の保証は観測可能な意味を変えない（MUST）．評価順（0003——引数は左から右）・effect row（0022 / 0023）・`throws`（0011）の規則はすべて不変である．本仕様は評価コストの保証のみを追加する．
- **T4（対象外）**: 次は本仕様の保証の対象外である：相互再帰（`f` → `g` → `f`），関数値・関数リテラル経由の呼び出し，trait メソッド経由の呼び出し（0020），自己呼び出しであっても末尾位置にないもの．これらの最適化は従来どおり規定しない（0003 不変）．

### Examples

保証される：自己呼び出しが本体 block の最終式にある（0046 の accept ループの骨格）：

```emela
fn serve_loop(server: Server) -> Unit uses { HttpServer } {
    try {
        let incoming = HttpServer.accept(server)
        HttpServer.respond(incoming, handle(incoming.request))
    } catch {
        e -> ()
    }
    serve_loop(server)  -- 末尾位置（T1）：スタックを消費しない（T2）
}
```

保証される：`if` の両 branch（カウントダウン）：

```emela
fn count_down(n: Int) -> Unit uses { Io } {
    if n == 0 {
        Io.print("done\n")
    } else {
        Io.print(n)
        count_down(n - 1)  -- 末尾位置（T1: if の else branch の最終式）
    }
}
```

保証されない：呼び出し結果に後続の演算がある（末尾位置ではない）：

```emela
fn fact(n: Int) -> Int uses {} {
    if n == 0 {
        1
    } else {
        n * fact(n - 1)  -- `*` の被演算子：末尾位置ではない（T1）
    }
}
```

保証されない：`try` block 内の自己呼び出し（catch フレームが残る）：

```emela
fn retry_forever() -> Unit uses { Http } {
    try {
        let _res = Http.get("https://example.com/health")
        retry_forever()  -- try block 内：末尾位置ではない（T1）
    } catch {
        e -> retry_forever()  -- catch arm の最終式：末尾位置（T1）
    }
}
```

（`try` 内の呼び出しは周回ごとにスタックを消費しうる．無限リトライは catch arm 側，または上の `serve_loop` のように「try / catch を済ませてから自己呼び出し」の形で書く．）

### Compilation Notes

この節は非規範的な実装上の補足である．

- **WASM backend**: 関数本体を `loop` block で包み，自己末尾呼び出しを「引数 local の再代入 + 先頭への `br`」に変換する．WASM の tail-call 拡張（`return_call`）を要しない．wasmi / WAMR の対応状況に依存しないのが T4 の限定（自己呼び出しのみ）の実装上の理由である．
- **JS backend**: 同じ変換を `while (true)` + 引数変数の再代入で行う．
- **判定**: 末尾位置（T1）は構文上の性質であり，型検査後の lowering で判定できる．`throws` を持つ関数の自己末尾呼び出しも，呼び出しが末尾位置にある限り（`?` が付くと T1 により対象外になる点に注意——`f(...)?` は末尾ではない）変換できる：error 値は変換後もそのまま関数の error 出口へ流れる．

### 他仕様との関係

- **0003（Function Definitions and Calls）**: 「末尾再帰呼び出しが最適化されるかどうかは規定しない」の一部（直接自己再帰・末尾位置）を本仕様が規定する．それ以外（T4）は不変であり，0003 の当該箇所には本仕様への注記を追記する．
- **0002 / 0004**: 全束縛不変・block 式の意味は不変．本仕様がループ構文の導入を不要にする．
- **0015 / 0005 / 0011**: `if` / `match` / `try`–`catch` の末尾位置の定義（T1）はこれらの式構造に基づく．意味論は不変．
- **0022 / 0023**: effect row の規則は不変（T3）．
- **0046（HTTP Server）**: 本保証の最初の利用者．`serve_loop` 形の無限 accept ループが規範例になる．
- **0000（Language Design Principles）**: WAMR 制約（スタック切り替え不可・小フットプリント）と整合する範囲（局所変換で実装可能な範囲）に保証を限定している．

### Open Questions

- 相互再帰への拡張（WASM tail-call 拡張の採用可否と連動．wasmi / WAMR の対応が揃えば T4 を緩められる）．
- 末尾位置の自己呼び出しであることを表明・検査する属性（`@tailcall` 等，0039 の閉集合への追加）を導入するか——「末尾のつもりが末尾でない」事故（`try` 内・`?` 付き）を宣言的に検出できる．
- catch arm 内の自己呼び出し（上例 `retry_forever`）で，`try` block 側の非末尾呼び出しと組み合わさった場合のスタック挙動の注意喚起を診断（lint，0035）にするか．
