# 0034: Multiline Expressions and Trailing Commas

Status: Draft

括弧内の改行と末尾カンマを定義する文法拡張．現行文法では改行が文・アイテムの区切りであるため，
`(...)` / `[...]` の内側で改行するとパースエラーになる．本仕様は **`(` `)` `[` `]` の内側で改行を
非有意化**し，**すべてのカンマ区切りリストで末尾カンマを許可**する．正規フォーマッタ（0035）が
最大行幅超過時に式を複数行へ機械的に展開するための前提であり，同時に手書きコードの折り返しと
diff の安定化（要素追加が 1 行の追加になる）にも資する．

## Summary

- `(` `)` および `[` `]` の内側では，改行はトークンとして現れない（字句レベルで定義）．
- ただしその内側でも `{` はブロックを開き，改行の有意性を復元する．`foo(match x { ... })` の
  ブロック内改行は従来どおり文・アームの区切りである．
- すべてのカンマ区切りリストで，閉じ記号直前の末尾カンマを許可する．
- それ以外の位置における改行の有意性は不変である．

## Motivation

Emela はセミコロンを持たず，改行が文とアイテムを区切る（0004 / 0010）．この設計は 1 行 1 文の
読みやすさに寄与するが，現行実装では括弧の内側でも改行が区切りトークンとして現れるため，引数
リストや配列リテラルを複数行に書けない．関数シグネチャや呼び出しが長くなったとき，折り返しの
手段が存在しないことになる．

0035 は最大行幅 100 桁を正規スタイルとして強制する．幅を超えた構文は機械的に複数行へ展開される
必要があり，その展開形が合法な文法でなければならない．括弧内改行の非有意化は Go（セミコロン
自動挿入の抑止規則）や Python（括弧内の暗黙の行継続）と同型の，改行有意言語における標準的な
解法である．

末尾カンマは，複数行リストの各要素行を同形にする（要素の追加・削除・並べ替えが常に 1 行の編集で
済み，diff が最小になる）．enum バリアントと match アームでは改行区切りが既に正規形であるため
（0005），末尾カンマの主対象は括弧・角括弧内のリストと effect 行である．

## Specification

### 括弧内の改行

- **G1**: `(` または `[` の内側（対応する `)` / `]` まで）では，改行は Newline トークンとして
  現れない (MUST)．空白と同様に読み飛ばされる．
- **G2**: G1 の内側でも，`{` から対応する `}` までの間では改行は従来どおり有意である (MUST)．
  この規則はネストに対して対称に適用される：字句解析器は開き記号のスタックを保持し，改行の
  有意性はスタック頂が `(` / `[` のとき非有意，それ以外（空または `{`）のとき有意と定義される．
  ただし例外として，effect 行 `uses { ... }` の braces はブロックでなくリスト括弧であり，
  その内側の改行は非有意である (MUST)．カンマは effect 行の唯一の区切りのままである．
- **G3**: G1/G2 以外の位置における改行の有意性は本仕様の前後で不変である (MUST)．すなわち
  ブロック内の文の区切り（0004），トップレベルのアイテムの区切り（0010），enum バリアント（0005）
  および match / catch アームの区切りは従来どおり改行（またはカンマ）である．
- **G4**: 対応の取れない閉じ記号はスタックを空のまま扱い，字句解析はエラーとせず継続してよい
  (MAY)．括弧の対応の検証は構文解析の責務である．

### 末尾カンマ

- **T1**: 以下のカンマ区切りリストにおいて，閉じ記号の直前に高々 1 つの末尾カンマを許可する
  (MUST)：
  - 呼び出し引数 `f(a, b,)`
  - 配列リテラル `[1, 2,]`
  - 仮引数リスト `fn f(x: Int, y: Int,)`
  - 関数型のパラメータ型リスト `(Int, Int,) -> Int`
  - enum バリアントの payload 型リスト `Pair(Int, Int,)`
  - パターンのフィールド束縛 `Pair(a, b,)`
  - 型パラメータリスト `fn id<T,>(x: T)`
  - 型引数リスト `List<Int,>`
  - effect 行 `uses { Stdout, Clock, }`
- **T2**: 要素が 0 個のリストにカンマだけを置くことはできない (MUST NOT)．`f(,)` は不正である．
- **T3**: 末尾カンマの有無は意味に影響しない (MUST)．

## Examples

```emela
-- G1: 引数リスト・配列は複数行に書ける
fn transfer(
    source: AccountId,
    destination: AccountId,
    amount: Int,
) -> Unit throws TransferError uses { Ledger } {
    ...
}

let matrix = [
    [1, 2, 3],
    [4, 5, 6],
]

-- G2: 括弧の内側でもブロック内の改行は文の区切りのまま
let result = apply(
    fn (x: Int) -> Int {
        let doubled = x * 2
        doubled + 1
    },
    seed,
)
```

## Compilation Notes（non-normative）

参照実装では G1/G2 を字句解析器のみで実装できる：開き記号のスタックを保持し，`\n` に遭遇した
とき，スタック頂が `(` / `[` であれば Newline トークンを発行しない．構文解析器は括弧内で
Newline を見なくなるため変更を要しない．T1 は各リストのカンマ消費ループで，カンマの直後に
閉じ記号が続く場合にループを抜ける 1 分岐を足すだけである．
